改善程序技术与设计思维的50个有效做法（下）

2020-11-17

类与函数之实现

条款29 ：避免传回内部数据的handles

指针并不是“传回内部资料handle”的唯一途径，references也很容易被误用。当函数传回一个“handle”时，而此handle所代表的信息应该不可被函数调用者所见，这个handle给了调用者无限制取用“私有字段data所指母标之物”的权利。

const member functions并不是唯一需要担心“传回handles”这档子事的函数。即使是non-const member functions，也必须面对这个事实：handle的有效性将它所对应的那个对象终了时一并期满。这可能会比client所预期的还快，尤其是当此对象是一个由编译器产生的暂时对象时。

因此，const member functions传回handles是不好的行为，因为它违反了抽象性。甚至对non-const member functions而言，传回handles也会导致麻烦，特别是当涉及暂时对象时，Handles可能变成空悬的，就像指针一样。而正如你努力避免所谓的dangling pointer一样，你也应该尽力避免所谓的dangling handles。

还是一样，没有理由太过绝对化。我们不太可能在一个稍微优点复杂的程序中挑出所有可能的dangling pointers，我们也不太可能消除所有可能的dangling handles。不过，尽管如此，如果你尽可能避免传回handles（除非有绝对必要），你的程序会受益。

条款30 ：避免写出member functions，传回一个non-const pointer或reference并以之指向较低存取层级的members

member function personAddress会将Person object内含的Address object提供给调用者，但或许是为了效率着想，传回值以by reference方式而非by value方式传递。不幸的是，这个member function使得“person::address成为private”的目标受到了挫败。

现在全局对象addr成为了scott.address的另一个名称，你可以利用它来任意读写scott.address。因此，scott.address不再是private,它变成了public,而这类存取层级的晋升,乃因member function personAddress之故。
reference不是唯一需要关心的对象，指针也可以玩这场游戏。然而，面对指针，你必须担心的不只是data members，还有member functions。因为函数可能传回一个pointer to member function。

尽管如此，或许有一天你得面对这样一种形势：为了达到效率上的规范，你必须写个member function，传回一个reference或pointer，指向一个较低存取层级的member。在此同时，你又不想牺牲private和protected所提供的存取保护。这种情况下你可以传回一个pointer或reference，指向一个const object，于是同时达成两个目标。

条款31 ：千万不要传回“函数内local对象的reference”或“函数内以new获得的指针所指的对象”

考虑第一种情况：“传回一个reference，指向一个local对象”。这里的问题是，local就是local，它们是区域性的，这意味着当它们被定义时就构造完成，当它们离开scope时就被析构。至于它们的scope，就是它们所在的那个函数。当函数返回时，控制权退离函数，所以此函数的local对象会被自动析构。如果你传回一个reference，指向函数中的一个local对象，则那个local对象在函数调用者重新获得控制权之前就已经被析构掉了。

当调用new而不使用local对象时，这种做法的确解决了local对象离开scope的会被析构的问题。但却导入了一个新问题。为了避免你的软件发生内存泄露，你一定得将delete实施于每一个以new获得的指针身上。问题是谁该为此函数内的new调用相应的delete呢？

条款32 ：尽可能延缓变量定义式的出现

亦即需要时才定义之

还记得吗，如果你定义一个变量，其型别有一个constructor或destructor，那么当控制权到达这个变量定义式时，你便得承受构造成本；当这个变量离开其scope时，你便得承受析构成本。意思是即使一个不曾被使用的变量，仍需耗费成本。这种情况应该尽可能避免。

你不只应该延缓变量的定义，直到非得使用该变量的前一刻为止，甚至应该尝试延缓这份定义，直到能够给予它初值引数为止。如果这样，不仅可以避免构造（析构）非必要的对象，还可以避免无意义的default constructions。

只要延缓变量的定义，你便可以改善程序的效率，增加程序的清晰度，降低变量意义的说明需求。看来似乎是该对那些“一开始出现”的变量定义说”拜拜”的时候了。

条款33 ：明智地运用inlining

inline函数背后的整个观念是，将“对此函数的每一个调用行动”都以函数代码取代之。这么做会增加你的目标代码的大小。

inline与register一样，只是对编译器的提示、建议，并不是一个强制命令。

当编译器拒绝将函数inline时，理所当然，函数将与其他函数的调用动作相同。

首先，我们经常把inline函数定义也放在头文件中，这样多个编译单元可以包含该头文件并获得其中的inline函数定义。而对于非inline函数，若将定义放在头文件中，头文件被包含多次时则包含多份该函数的定义，在连接时发生错误。

若编译器拒绝将函数inline时，其定义在头文件中，然而其在被调用时视作非inline函数，则在连接时会出现错误。为此，旧规则规定，在未被inlined成功时，视作将此函数声明为static处理。此时，每个包含该函数的文件都有一个该函数的static副本。这将增加程序代码的体积。而新规则则只会生成一份函数副本。

同时，对一个成功inlined的函数进行取地址时，旧规则同样对每个编译单元都需要生成一个static副本，而新规则则只会生成一个。

对于构造函数、析构函数，其包含调用数据成员的构造函数与析构函数，只是这些代码由编译器来完成。

另外，对于inline函数，当改动函数时，所有用的该函数的程序都必须重新编译。而对于非inline函数则只需要重新连接即可（甚至采用动态连接库可以不知不觉地实现）。通常，含有static对象的函数避免被声明为inline。

最后，inline函数出错难以定位。无法设置断点、next等方式。

总之，不要随意使用inline，除非那些非常普通简洁的函数，以及那些程序中占重要效率地位的函数。

条款34 ：将文件之间的编译依赖关系降至最低

假如你在修改程序，只是修改了某个class的接口的实现，而且修改的是private部分。之后，你编译时，发现好多文件都被重新编译了。这种问题的发生，在于没有把“将接口从实现中分离”。

分离的关键在于“声明的依存性”替换了“定义的依存性”，这正是编译依存性最小化的本质：现实中，让头文件尽可能自我满足，万一做不到，则让它与其他文件内的声明（不是定义）相依。其他都源于以下设计策略：

  • 如果使用object references或object pointers可以完成任务，就不要使用object。因为，使用references或pointers只需要一个声明，而定义objects需要使用该类型的定义。

  • 如果可以，尽量以class声明式替换class定义式。但是，当声明函数使用某个class时，即使是by value方式传递该类型参数/返回值，都不需要class定义。但是在使用这些函数时，这些classes在调用函数前一定要先曝光。客户终究是要知道classes的定义式，但是这样做的意义在于：将提供class定义式（通过#include完成）的义务，从函数声明所在头文件，转移到函数调用的客户文件。

  • 为声明式和定义式提供两个不同的头文件。程序中，不应该让客户给出前置声明，程序作者一般提供两个头文件，一个用于声明式，一个用于定义式。在C++标准库的头文件中, < iosfwd> 内含iostream各组件的声明式，其对应定义分布在不同文件件，包括< sstream>,< streambuf>,< fstream>,< iostream>。

因此，避免大量依赖性编译的解决方案就是：在头文件中用class声明外来类，用指针或引用代替变量的声明；在cpp文件中包含外来类的头文件。

继承关系与面向对象设计

条款35 ：确定你的public inheritance，模塑出“is a”的关系

以C++完成面向对象程序设计，最重要的一个规则是：public inheritance（公开继承）意味着“是一种（isa）”的关系。

如果你令class D以public形式继承了class B，你便是告诉C++编译器说，每一个型别为D的对象同时也是一个型别为B的对象，反之并不成立。你的意思是B比D表现出更一般化的概念，而D比B表现出更特殊化的概念。你主张“B对象派的上用场的任何地方，D对象也一样可以派上用场”，因为每一个D对象都“是一种”B对象，反之如果你需要一个D对象，B对象无法效劳。

根据生活经验我们知道，每个“学生”都是“人”，但并非每个“人”都是“学生”。这便是上述继承体系所主张的意义。

在C++领域中，任何函数如果期望获得一个型别为Person的引数，都可以接收一个Student对象而无任何异议。这个论点只有对public inheritance 才成立。

条款36 ：区分接口继承和实现继承

最明确而直接易懂的概念（也就是public inheritance）已经出现，但是再靠经一些，你会发现它们分为两类：函数接口的继承和函数实现的继承。这两种继承的差异，很像书本导读所讨论的函数声明与函数定义之间的差异。

身为一个class设计者，有时候你会希望derived class只继承member function的接口；有时候你又会希望derived class同时继承member function的接口和实现，并能够以你所提供的实现代码来改写（override）它们；有时候你又会希望derived classes同时继承member function的接口和实现，并且不允许它们被改写（override）。

  • Member function的接口总是会被继承。public inheritance意味着“是一种（isa）”，所以任何事情只要对base class而言为真，就一定对其derived classes为真。因此，如果某个函数可施行于某个class上头，一定也可施行于其subclasses上头。
  • 声明一个纯虚拟函数的目的是为了让derived classes只继承其接口。
  • 声明一般（非纯）虚拟函数的目的，是为了让derived classes继承该函数的接口和缺省行为。
  • 声明非虚拟函数的目的是为了令derived classes继承函数的接口及其实现。

书上提倡用纯虚函数去替代虚函数，因为虚函数提供了一个默认的实现，如果派生类的想要的行为与这个虚函数不一致，而又恰好忘记去覆盖虚函数，就会出现问题。但纯虚函数不会，因为它从语法上限定派生类必须要去实现它，否则将无法定义派生类的对象。

同时，因为纯虚函数也是可以有默认实现的（但是它从语法上强调派生类必须重定义之，否则不能定义对象），所以完全可以替换虚函数。

普通函数所代表的意义是不变性凌驾与特异性，所以它绝不该在派生类中被重新定义。

在设计类成员函数时，一般既不要将所有函数都声明为non-virtual（普通函数），这会使得没有余裕空间进行特化工作；也一般不要将所有函数都声明为virtual（虚函数或纯虚函数），因为一般会有一些成员函数是基类就可以决定下来的，而被所有派生类所共用的。这个设计法则并不绝对，要视实际情况来定。

条款38 ：绝对不要重新定义继承而来的缺省参数值

让我们一开始就将讨论范围简化。缺省参数只能存在于函数之中，而你只能够继承两种函数：虚拟函数和非虚拟函数。因此，重新定义一个缺省参数值的办法就是重新定义一个继承而来的函数，然而重新定义一个继承而来的非虚拟函数永远是错误的行为，所以我们可以安全地将讨论局限于“继承一个带有缺省参数值的虚拟函数”。

再这种情况下，本条款的理由就非常直接而明确了：虚拟函数系动态绑定（dynamically bound），而缺省参数值却是静态绑定(statically bound)。

对象的静态型别，是程序声明它时所采用的型别。

对象的动态型别系指“对象目前所代表的型别”。一如其名称所示，可以在程序执行过程中改变。

虚拟函数系动态绑定而来，意思是这一类特殊函数之所以被调用，系取决于对象的动态型别。

当你考虑带有缺省参数值的虚拟函数时，问题来了，因为就如上面所说，虚拟函数是动态绑定的，而缺省参数值却是静态绑定的。意思是你可能会在调用一个定义于derived class内的虚拟函数的同时，却使用base class为它所指定的缺省参数值。

条款41 ：区分inheritance和templates

  • template应该用来产生一群classes，其中对象型别不会影响class的函数行为。
  • inheritance应该用于一群classes身上，其中对象型别会影响class的函数行为。

将这两点融入内在思想中，你便可以掌握inheritance和templates之间正确的选择之道。

条款42 ：明智地运用private inheritance（私有继承）

如果class之间的继承关系是private，编译器通常不会自动将一个derived class object转换为一个base class object。这和public inheritance的情况相反。

第二条规则是，由private base class继承而来的所有members，在derived class中都会变成private属性，纵使它们在base class中原本是protected或public属性。

private inheritance意味着“根据某物实现”。如果你让class D以private方式继承class B，这么做的原因是，你想采用已经撰写于class B内的某些程序代码，而不是因为B和D的对象有任何概念关系存在。所以private inheritance纯粹只是一种实现上的技术，与观念无涉。private inheritance意味着只有实现部分被继承；接口部分应略去。如果D以private方式继承B，意思是D对象根据B对象实现而得，再没有其他意思了。

和layering技术一样，根据private inheritance而完成的实现代码，可避免程序代码重复，因为型别安全（type-safe）的interface classes，其内除了以inline方式调用底层的函数之外，一无长物。

唯有通过继承，才得以取用protected members，唯有通过继承，才允许虚拟函数被重新定义。由于虚拟函数和protected members的存在，当我们欲表现classes之间“根据某物实现”这种关系时，有时候private inheritance是唯一具有实用价值的做法，一旦private inheritance成为你所能支配的实现技术中最适当的一个，你就不应该害怕使用它。然而在此同时我仍要强调，通常layering是最受喜爱的技术，所以你应该尽可能运用它。

条款43 ：明智地运用多继承（multiple inheritance，MI）

C++的多继承有一个毋庸置疑的事实就是：它打开了潘多拉的盒子，释放出一大堆单继承之中并不存在的复杂性。其中最根本的一个复杂性就是模棱两可。如果derived class从一个“以上”的base classes继承了同一个member名称，那么对该名称的任何取用行为都会造成模棱两可的问题；你必须明白说出你要的是哪个member。

  • 多继承比单继承复杂，引入了歧义的问题，以及虚继承的必要性；
  • 虚继承在大小、速度、初始化/赋值的复杂性上有不小的代价，当虚基类中没有数据时还是比较合适的；
  • 多继承有时也是有用的。典型的场景便是：public继承自一些接口类，private继承自那些实现相关的类。

总结

我们应当将多继承视为面向对象设计工具箱中一个有用的工具。相比于单继承它会更加难以理解， 如果有一个等价的单继承设计我们还是应该采用单继承。但有时多继承确实提供了清晰的、可维护的、合理的方式来解决问题。 此时我们便应该理智地使用它。

  • 多继承比单继承复杂，引入了歧义的问题，以及虚继承的必要性；
  • 虚继承在大小、速度、初始化/赋值的复杂性上有不小的代价，当虚基类中没有数据时还是比较合适的；
  • 多继承有时也是有用的。典型的场景便是：public继承自一些接口类，private继承自那些实现相关的类。

条款44 ：说出你的意思并了解你所说的每一句话

“说出你的意思”只是战斗的一半而已。另一半就是要“了解你所说的每一句话”，这两点的重要性不相上下。例如，如果你贸然便将member functions声明为非虚拟，而没有认识到这么做就是在subclasses身上设了枷锁，那么你的行为便是不负责任又不道德。声明一个非虚拟member function的意思是，这个函数的不变性凌驾于其变异性之上。

public inheritance和isa之间的对等关系，以及nonvirtual member functions和“不变性凌驾于变异性之上”之间的对等关系，只是“C++语言结构”与“C++设计观念”之间众多映照实例中的两个。以下摘列最重要的一些观念映对。

  • 共同的base class意味着共同的特性。如果class D1和class D2都声明class B为base，D1和D2便从B手中继承了共同的data members和共同的member functions。
  • Public inheritance（公开继承）意味着“是一种（isa）”。如果class D以public方式继承class B，每个D对象便也是一个B对象，反之则不成立。
  • Private inheritance（私有继承）意味着“根据某物实现(is-implemented-in-terms-of)”。如果class D以private方式继承了class B，D对象便是根据B对象实现；B对象和D对象之间并没有任何概念上的关系。
  • Layering意味着“有一个（has-a）”或“根据某物实现(is-implemented-in-terms-of)”。如果class A内含一个型别为B的data member，那么A对象之中便有一个型别B的成分，或说A对象系根据B对象实现。

只有当牵涉到public inheritance（公开继承）时，以下数点才成立:

  • 纯虚拟函数意味着：只有其函数接口会被继承。如果class C声明了一个纯虚拟函数mf，则C的subclasses必须继承mf的接口，而C的具象subclasses必须提供自己的实现代码。
  • 一般(非纯)虚拟函数意味着：函数的接口及缺省实现代码都会被继承。如果class C声明了一个一般(非纯)虚拟函数mf，C的subclasses必须继承mf的接口，它们也可以继承mf的缺省实现代码。
  • 非虚拟函数意味着：此函数的接口和其实现代码都会被继承。如果class C声明了一个非虚拟函数mf，C的subclasses必须同时继承mf的接口和实现代码。事实上可以说，mf为C定义了一个“不变性凌驾于变异性之上”的性质。

杂项讨论

条款45 ：清楚知道C++（编译器）默默为我们完成和调用哪些函数

对于一个空类，编译器可能默默生成下面四个函数。

  • 只有在这些函数被需要、被调用时，编译器才会默认生成：构造、拷贝构造、析构、拷贝赋值这四个函数。
  • 默认生成的构造、析构函数被用来调用“隐藏”的代码，如基类的构造、析构函数
  • 默认析构函数是 non-virtual，除非继承自父类的 virtual 析构函数
  • 当存在构造函数时，编译器不会再自动生成无参默认构造函数
  • 编译器的默认 copy 构造函数，会把每一个 non-static 成员变量拷贝到目标对象

当情况不合适时，编译器拒绝生成默认函数，如下列情况：

  • 当含有 reference 成员变量
  • 当含有 const 成员变量
  • 当基类 copy assignment 函数为 private 时。因为默认生成的 copy assignment 函数想调用基类的 copy assignment 函数而不得。

条款46 ：宁愿编译和连接时出错，也不要执行时才错

这个设计结合了数个概念。

  • 第一:Month构造函数私有,可以阻止clients产生新的月份.
  • 第二:每个Month object都是const,所以它们不可能被修改。
  • 第三:获得Month object的唯一途径调用函数,不用担心Month object没有被初始化.

这个例子以”编译期检验”取代了”执行期检验”

消除所有的执行时期检验是不切实际的,尽管如此,将执行时期的检验移到编译期或连接期,还是一个值得的目标。

条款47 ：使用non-local static objects之前先确定它已有初值

当一个编译单元内某对象的初始化动作，与另一个编译单元内某对象的值相依，而后者也需要初始化。
可能导致第一个对象没有初值。

解决的办法：Singleton pattern。
将每一个non－local static object转到一个专属函数中，接下来令函数传回一个reference，指向内含的那个static object。这样，客户不再直接取用static object，而是改调用函数。

依据：C++明确指出函数中的static对象的初始化时机：在函数调用期间第一次遇到对象的定义时。

这样做的另一个好处：如果从未调用带有非本地静态对象的函数，那么绝不会付出对象的构造和析构成本。

客户端几乎完全不变，只不过它们现在调用theFileSystem()，而不是直接取用theFileSystem。

这项技术之所以有效，是因为它为你的对象带来了一个合理的初始化次序。如果你要求对象A一定得在对象B之前先初始化，而你又让A的初始化动作取决于B是否已被初始化，你便有麻烦了。坦白地说，这是你自找的。

条款49 ：尽量让自己熟悉C++标准程序库

标准库的每一样东西几乎都在namespace std中，然而世界上有无可计数的C++程序代码依赖那些已经使用多年的“准“标准程序库，例如声明于< iostream.h>,< limits.h>等文件中的功能。

解决方法，设计新的头文件名，给披上std外衣的各个组件使用。

  • 将原有的c++头文件中的.h扩展拿掉，于是< iostream.h>变为< iostream>。
  • C头文件采用相同办法，并在最前面加上一个字母c。于是< stdio.h>变为< cstdio>

关于标准程序库，必须知道的事情：几乎其中每一样东西都是template。

条款50 ：加强自己对C++的了解

C++涵盖的东西很多：有C，有overloading（重载）、有object-oriented（面向对象）、有template，有exception、有namespace。这么一大堆素材，有时候让人喘不过气来。你该如何面对这堆东西？

  • 与C兼容。
