改变旧有的C习惯

条款2 ：尽量以取代<stdio.h>

是的，它们是可移植的。是的，它们是高效率的。尽管如此，scanf和printf及其家族都还不够完美。尤其是它们都不具型别安全（type-safe）性质，也都不可扩充。由于型别安全性和扩充性正是C++的基石之一，恐怕你最好是在此刻选择舍弃scanf和printf。

iostream程序库中的各个classes和各个函数所提供的型别安全性和扩充性，比你最初想想的还要有用，所以不要只因为你目前还在使用<stdio.h>，就将iostream视而不见。

从技术上来说，已经没有所谓的<iostream.h>。当标准委员会删除其他 non-C标准头文件的扩展名时，就消灭了它。

条款3 ：尽量以new和delete取代malloc和free

malloc和free（以及它们的各式变形）带来的问题很简单：它们对constructors（构造函数）和destructors（析构函数）一无所知。
由于new和delete能够与constructors和destructors有适当的互动，所以它们是比较好的选择。

当你企图对着一个“从new而得的指针”调用free时，或是对着一个“从malloc而得的指针”调用delete时，其结果都未有定义。只要你确定以malloc获得的指针总是以free释放之，以new获得的指针总是以delete释放之即可。

内存管理

条款5 ：使用相同形式的new和delete

以下动作有什么错？

每件事情看起来都井然有序。使用了new，也搭配了对应的delete。但还是有某样东西完全错误了：你的程序行为未定义。由stringArray所指的100个string对象中的99个未必能够被适当地删除，因为它们的destructors没有被调用。

游戏规则很简单：如果你在调用new时使用了[]，则你必须在调用delete时也使用[]。如果你在调用new的时候没有使用[]，那么你也不应该在调用delete时使用[]。

条款6 : 记得在destructor中以delete对付pointer members

大部分时候，classes如果要动态分配内存，会在constructors中以new配置内存，并于稍后在destructor中以delete释放内存，当你初次写这个class时，要做这些动作并不难，当然，前提是你必须记得对所有“在constructor中被分配以内存”的members使用delete。

每在classes中加上一个pointer member时，几乎总是需要配合的以下没意见事情：

  • 在每一个constructors中将该指针初始化。如果没有任何一个constructor会将内存配置给该指针，那么指针应该初始化为0.
  • 在assignment运算符中将指针原有的内存删除，重新配置一块。
  • 在destructor中删除（delete）这个指针。删除一个null指针是安全的（什么也没做）。

条款7 ：为内存不足的状况预做准备

当operator new没有能力配置出你所需要的内存时，会丢一个exception给你。

那么如何处理呢？如果你打算建立一个简易的错误处理策略，你可以要求当内存需求无法获得满足时，让系统调用你所指定的一个函数。这样的策略依赖一个公约：当operator new无法满足需求时，它会在抛出exception之前调用一个client专属的错误处理函数，此函数通常称为 new-handler。

一个良好的new-handler函数必须完成以下事情之一：

  • 让更多内存可用。这或许能够让operator new的下一次内存配置行为成功。实现此策略的方法之一就是在程序起始时配置一大块内存，然后在new-handler第一次被调用时释放之。

  • 安装一个不同的new-handler。如果目前的new-handler无法让更多内存可用，或许它知道另一个new-handler手上握有比较多的资源。

  • 卸除这个new-handler，也就是说，将null指针传给set_new_handler.一旦没有安装任何new-handler，operator new就会在内存配置失败时抛出一个型别为std::bad_alloc的exception。

  • 抛出一个exception，型别为std::bad_alloc（或其派生型别）。这样的exception不会被operator new 捕捉，所以它们会传送到最初提出内存需求的那个点上。

  • 不返回，直接调用abort或exit，这两者都是C标准函数库提供的函数。

这些选择让你在实现new-handler函数时握有极大的弹性。

C++并不支持“class专属的new-handler”，但其实我们也不需要它的支持；是的，我们可以自己完成这种行为。你只需要让每一个class供应自己的set_new_handler和operator new 即可。

不论你使用正常形式（也就是会抛出exception）的new，或是“nothrow”形式的new，重要的是你已经准备好处理内存配置失败的情况了，最简单的方法就是利用set_new_handler，因为它在两种形式中都可用。

条款8 ：撰写operator new 和operator delete时应遵行的公约。

当你决定撰写自己的operator new时，有一点相当重要：你的函数行为应该与缺省的operator new保持一致性。实际一点说，这意味着你的函数应该有正确的返回值，内存不足时调用一个错误处理函数，并准备应付“no memory”（不索求任何内存）的需求。你还得避免不经意间遮掩了“正常形式”的new。

传回值的部分很简单，如果你能够供应足够的内存，就传回一个指针指向它。如果你不够，则抛出一个型别为std::bad_alloc的exception。

此外，C++ standard要求，即使用户要求的是0 bytes内存，operator new也应该传回一个合法指针。

如果你的内存配置函数支持new-handler并正确处理了0sized申请动作，你就几乎完成了所有工作。至于你的内存释放函数，除了处理null指针外，还有另外一点点事要做。只要为这些函数的member版本再加上继承能力，像变魔术似的，你便完成了全部工作。

内部scope内的一个声明名称，会遮掩外部scope的相同名称，所以如果global scope和class scope内部都有函数f，那么member function f会遮掩掉global function f :

这并不让人惊讶，而且也不至于困惑，因为global function和member function总是以不同的语法形式被调用。

如果在class内声明一个operator new函数，你将因而不经意地妨碍程序调用new“正规”形式的能力。

解决方法之一是写一个class专属的operator new，并令他支持正规调用形式。

另一种做法是为operator new的每一个额外参数提供默认参数值。

条款10 ：如果你写了一个operator new，请对应写一个operator delete

为什么会有人想要撰写自己的一份operator new或operator delete呢？答案多半是为了效率。缺省版的operator new和operator delete对于一般目的而言，已是十分适当而足够，但它们的巨大弹性导致无可避免地需要在较为受限的环境下花费一些内存空间，以便提高效率。

由于缺省版的operator new是一个泛用型内存配置器，它必须能够配置任意大小的区块。同样道理，缺省版的operator delete必须能够释放由operator new所配置的任意大小的区块。为了让operator delete 知道需要释放多少内存，必须有某种办法让它知道operator new配置了多少内存。让operator new得以告诉operator delete说“它究竟配置了多少内存”的一个常见的办法就是，在它所传回的内存前端“加挂”一些额外数据，说明配置的区块大小。

重点在于operator new和operator delete需要一起合作，所以如果你写出了一个operator new，请确定也写了一个对应的operator delete。

构造函数、析构函数和Assignment运算符

条款11 ：如果class内动态配置有内存，请为此class声明一个copy constructor和一个assignment运算符

当你没有自声明assignment运算符时，C++产生一个默认的assignment运算符并调用之。这个默认的assignment运算符执行memberwise assignment动作（亦即一个member一个member地逐次设值），其实就是一个bitwise copy（一个位一个位的逐次拷贝）动作。当两个没有定义assignment运算符的类对象赋值时，该对象包含一个指针，指向数据成员：
b=a；
这样的状况至少存在两个问题，第一，b原先所指的内存没有被释放掉；它永远遗失了。这是一个典型的内存泄露问题。第二，a和b内含的指针如今都指向相同的字符串，因此，当其中一个离开生存空间时，其destructor会删除内存，而此内存目前仍然被另一个内存所指。

copy constructor的情况和assignment运算符的情况有一点不同，因为只要程序中有pass-by-value的动作，就会调用copy constructor。

如果你的class拥有任何指针，那么请撰写你自己的copy constructor和一个assignment运算符。

对于某些class二元，实现copy constructor和assignment运算符会有点得不偿失，特别是当你有理由相信你的clents不会执行copy或assignment动作时。将这些函数声明位private，并且不要定义（实现）它们。这可阻止clients调用它们，也可以避免编译器产生它们。

条款12 ：在constructor中尽量以initialization动作取代assignment动作

亦尽量运用constructor的member initialization list

在构造函数中，你必须将参数值转设给对应的data members。有两种做法，第一种是使用所谓的member initialization list（成员初始化列表）。
第二种做法是在constructor函数内进行assignment动作。

从纯粹的程序眼光而言，许多时候一定得使用initialization list。事实上const members和reference members只能够被初始化，不能够被赋值。

在效率上来说，如果使用member initialization list，只有一个member function会被调用；如果在constructor内使用assignment动作，会调用两个member functions。经由member initialization list来进行初始化工作，绝不会比在constructor函数本体内使用assignment来的没效率；事实上通常是比较有效率的！

然而在一种情况下，以assignment取代initialization是合理的，那是“当你有一大堆内建型别之data members，而你希望在每一个constructor中以相同方式将它们初始化”的情况。

内建型别之non-const和non-reference对象，其initialization和assignment之间并没有任何操作上的差异。

注意：static class members绝不应该在一个class’s constructor中被初始化。static members在每一个程序执行过程中只应该初始化一次，所以企图在每次class object产生之际就为它们初始化一次是不合理的。

条款13 ：initialization list中的members初始化顺序应该和其在class内的声明次序相同

class members系以它们在class内的声明次序来初始化；和它们子member initialization list中出现的次序无关。

如果members系以它们出现在initialization list的次序来初始化，那么w1和w2的data members将以不同的次序来进行构造。对象的data members的destructor总是以“和其constructors相反的次序”被调用，因此，如果本段一开始的叙述为真，编译器就必须追踪记录每一个对象内的members的初始化顺序，以便确保能够以正确次序调用destructor。这将带来昂贵的成本。为了避免这样的额外负担，任何型别的所有对象，其构造和析构次序都相同，而members出现在initialization list中的次序将不予计较。

结论是：对象被初始化时，如果你希望确实掌握真正发生了些什么事，请以class内的members声明次序，将各个members列于initialization list中。

条款14 ：总是让base class拥有virtual destructor

将destructor声明为virtual可以确保其行为有良好定义，派生类的析构函数和基类的析构函数会良好的被调用。

如果class并未含有任何虚拟函数，往往意味着它其实并不是要被当作base class 使用。当class不企图成为一个base class时，令其destructor为virtual通常是个坏主意。

条款17 ：在operator=中检查是否“自己赋值给自己”

“自己赋值给自己”可以发生在看起来比较良性的形式中：
a=b；
如果b是a的一个别名（例如b是一个reference，初值为a），那么这也是“自己赋值给自己”，虽然表面上看起来不像！这是aliasing（别名，同一对象拥有两个或更多个名称）的例子。

两个理由促使我们在assignment运算符中特别谨慎地处理可能发生的别名问题。比较次要的理由是为了效率。如果你能够在assignment运算符中一开始就侦测到“自己赋值给自己”的行为，你可以立刻返回，这或许能节省许多不必要的工作。

不过，侦测“自己赋值给自己”行为，更重要的理由是要确保正确性。记住，assignment运算符在为其左侧对象配置新资源之前，通常必须先将原有配置给左侧对象的所有资源释放掉。然而如果是“自己赋值给自己”，这个资源释放动作可能会造成惨重的灾情，因为旧资源可能在配置新资源的过程中派上用场。

辨识两个对象是否为同一个（也就是有相同的身份），方法之一就是看看它们是否有相同的值。
另一个“对象等同”的定义是，看看它们的内存地址是否相同。
这对许多程序而言已经足够。

如果你需要一个更精巧的机制用以判断两个对象是否相同，你就得自己实现。

别名（aliasing）问题和对象等同（object identity）问题并不限于只在operator=函数内出现。那只是你可能遭遇这类问题的一个场所而已。只要出现references和pointers，任何两个代表兼容型别的对象名称都可能实际上指向同一对象。

类与函数之设计和声明

只要在程序中声明一个新的class，便是产生一个新的型别（type）。所以class的设计就是一种型别设计。

条款18 ：努力让接口完满(complete)且最小化

class的所谓客户端接口，是一个开放给class用户并为他们所接收的接口。一般而言，接口之中只有函数，因为如果其中也有数据，会导致许多缺点。

所谓”完满”接口，即允许clients做他们可以合理要求的任何事情。也就是说，对于client可能希望完成的任何合理工作，class都应该提供一个合理的方法来完成，即使不像clients所喜欢的那么方便。至于一个“最小化”接口，则是尽可能让函数个数最少，不至于有任何两个member functions功能重叠。

大型接口容易令人迷失。
大型class接口的第二个缺点是可维护性较差，class内含的函数愈多，维护和强化的难度愈高，反之则愈低。
最后一点，长长的class定义，会导致长长的头文件。由于头文件往往必须在程序每次编译的时候被读入，如果class定义式比必要的长度还长，会导致编译时间过长，影响整个程序的开发。

无条件地为接口增加函数，并非不需要成本。所以你得谨慎思考，是否一个新函数的便利性可以平衡其额外的复杂性、可理解性、可维护性，以及编译速度所带来的成本。

条款19 ：区分member functions，non-member functions和friend functions三者

member functions和non-member functions之间的最大差异是，member functions可以是虚拟函数而non-member functions不可以。所以如果你有一个函数必须动态绑定，你必须对它动用虚拟函数，而此虚拟函数必须是个class member。

假设f是你企图声明的一个函数，C是个class，与f有观念上的关联：

  • 虚拟函数必须是class members。如果f必须是虚拟函数，那么就让它成为C的一个member function。
  • 绝不要让operator>>和operator<<成为members。如果f是operator>>或operator<<，请让f成为一个non-member function；如果f同时还必须取用C的non-public members，请让f成为C的一个friend。
  • 只有non-member functions 才能在其最左端引数身上实施型别转换。如果f需要在其最左端引数身上施行型别转换，请让f成为一个non-member function；如果f同时还必须取用C的non-public members，请让f成为C的一个friend。
  • 上述之外的每一种情况，都应该设计成member function。如果不适用其他情况，就让f成为C的一个member function吧。

条款20 ：避免将data members 放在公开接口中

首先，让我们从“一致性”的角度来看这个题目。如果公开接口中的每样东西都是函数，clients就不需要搔首晃脑地试图记住“存取你的class的某个member时”是否需要使用小括号。

使用函数，可以使你更精确地掌控data members的可存取性。如果你让一个data member成为public，则每个人都可以直接读写它。但如果你通过函数存取其值，你可以实现出“不处理”、“只读处理”、“可读可写”三种形式。

结论就是，把data members置于公开接口，等于是自找麻烦。为了安全起见，请将你的所有data members隐藏在函数抽象性质所形成的一道保护墙后面。如果你现在就这么做，便可以获得“一致性”，以及优良的存取控制，不需要耗费额外的成本！

条款21 ：尽可能使用const

const所带来的曼妙事情是，它允许你以某种语意上的规范表示“某个对象不应该被修改”，而编译器将严格奉行这个规范。

关键词const的用途很广。在class之外，你可以用它来定义global常量或namespace常量以及static对象。而在classes内，你可以将它使用于static/nonstatic data members身上。

面对指针，你可以明确指定指针本身为const，或是指针所指的资料为const，或是两者皆为const，或是两者皆否：

const最具威力的用途是在函数声明时派上用场。在一个函数声明式中，const可以用来修饰函数的传回值、个别参数，以及整个函数。最后一项只对member functions才成立。

member functions即使只是常量性有所不同，也可以重载，这是C++的一个重要特性。

将一个member function声明为const，真正的意义到底是什么呢？有两个流行的论点：bitwise constness（位常量论）和conceptual constness（概念常量论）。

bitwise constness（位常量论）阵营认为，如果一个member function不修改对象任何data members，它便是一个const member function。事实上，bitwise constness（位常量论）正是C++对于所谓“常量”的定义，因此一个const member function不允许修改其调用者的任何data members。

conceptual constness（概念常量论），这个观念主张const member function可以更改其调用者（某个对象）内的某些位，但是只能在不被clients侦测到的情况下为止。

关键字mutable，这是C++标准委员会深思熟虑后推出的一个解答，施行于nonstatic data members身上，可以有效解放它们在bitwise constness方面的束缚。将data members成为mutable时，data members可在任何地点被更改，甚至即使在一个const member functions内。

还有一种情况，常量转型可能有用而且安全，那就是当你有一个const对象时，想要传递给一个接收non-const参数的函数，而你知道这个参数在函数中并不会被更改。第二个条件很重要，因为将一个只读对象的常量性转换掉，永远是安全的，纵使该对象最初定义为const。

条款22 ：尽量使用pass-by-reference，少用pass-by-value

C语言中的每样东西都是passed by value，C++承担这项传统的做法是：把pass-by-value当作缺省行为。除非你另外指定，否则函数参数都是以实参的副本作为初值，而调用端所获得的亦是函数传回值的一个副本。

对象以by value的方式传递，其实际意义是由该对象的copy constructor决定的。这可能会使pass-by-value成为成本很高的动作。

当对象里的函数传值两次时，一次用于参数，一次用于传回值，当调用此函数一次，全部的成本是12个constructor和12个destructor！

当以by reference方式传递时，这种做法的效率就高多了：没有任何constructor或destructor会被调用，因为没有必要产生任何新对象。

以by reference的方式传递参数，有另一个优点：可避免所谓的“切割(slicing)”问题。当一个derived class object被交出去当作一个base class object时，它原本所以“成为一个derived class object”的所有特征，都会被切割掉，只留下内部一个base class object。

解决切割问题的方法，就是以by reference方式传递参数。

references的底层几乎都是以指针完成，所以passing by reference通常意味着传递的是指针。如果你有个小对象，例如一个int，那么pass by value可能比pass by reference的效率更高一些。

条款23 ：当你必须传回object时，不要尝试传回reference

在坚定追求pass-by-reference的纯度中，常常犯下一个致命错误：它们开始传递一些reference，指向其实并不存在的对象。这不算好事情。

所谓reference只是一个名称，一个既有对象的名称。任何时候看到一个reference，你都应该立刻问自己，它的另一个名称是什么，因为它一定是某物的另一个名称。

像operator*这样的函数传回一个reference只是在浪费时间。

绝不要返回指向本地栈对象的指针或者引用，指向堆对象的引用，或者在有可能需要多个对象的时候返回指向本地静态对象的指针或者引用。

条款24 ：在函数重载和参数缺省化之间，谨慎选择

函数重载和参数缺省化之间所引起的混淆，系基于一个事实：它们都允许以不同的形式调用同一个函数名称：

我们应该使用哪一种做法呢？

答案视另外两个问题而定。第一，是否有适当的值可以用来当作缺省参数？第二，你希望使用多少个算法？一般而言，如果你可以选择一个合理的默认值，而且你只需要一个算法，那么最好是使用缺省参数法。否则你可以使用重载函数。

假设你要写一个函数，计算最多5个整数的平均值。你不应该使用缺省参数值，因为函数的结果视参数个数而定。

另一种必须使用重载函数的情况是：你得完成一项特别任务，而采用的算法视输入而定。这种情况常常发生在constructors身上：default constructor可以无中生有地构造出一个对象，copy constructor则从既有对象身上构造出另一个对象。

条款26 ：防卫潜伏的ambiguity（模棱两可）状态

C++也有一个哲学信仰：它相信潜在的模棱两可状态不是一种错误。

编译器看到对f的调用动作，于是知道必须接获一个A object，即使它实际到手的是个B object。有两个一样好的办法可以完成这件事，其中之一是调用class A constructor,以b为引数构造出一个新的A object。另一个方法是调用class B的转换函数，将b转换为一个A object。这两个方法一样好，编译器会拒绝在它们之间表态，于是造成模凌两可的现象。

多继承是“潜伏期模凌两可”的好发因子。最直接易懂的情况就是“derived class由一个以上的base class出继承获得相同的member名称”。

既然写程序和函数库时有这么多不同的情况会产生潜在的二义性，那么，一个好的软件开发者该怎么做呢？最根本的是，一定要时时小心它。想找出所有潜在的二义性的根源几乎是不可能的，特别是当程序员将不同的独立开发的库结合起来使用时，但在了解了导致经常产生潜在二义性的那些情况后，你就可以在软件设计和开发中将它出现的可能性降到最低。

条款27 ：如果不想使用编译器暗自产生的member functions，就应该明白拒绝它

对类中如果不自己定义编译器会为你自动生成的函数，解决之道是将它们声明为private。是的，只要明白声明这个函数，你就可以阻止编译器自动生成一个版本；而只要让此函数成为private，你就可以不让别人调用它。

然而这个办法并不安全，因为member functions和friend functions还是可以调用你的private函数。除非你心里头明白，压根就不要定义它，那么一来如果你不经意调用这个函数，连接时便会出错。

条款28 ：尝试切割global namespace（全局命名空间）

global scope的最大问题就是，这家伙是唯一的。在一个大型软件项目中，常会有一群人在此唯一的scope中放置各种符号名称，而这往往导致名称的冲突。

一个较好的解决办法是使用C++ namespace。所谓namespace（名称空间），其本质其实只是一个比较漂亮的方法，让你使用你所知道的前导词而又不需要老是出现那么又臭又长的名称。

clients可以以三种方式取用你的namespace内的符号：一是将namespace内的所有符号汇入一个scope内

二是将个别符号汇入一个scope内。

三是每次使用符号时明确地加以资格修饰。

namespace有一件最棒的事情就是，潜伏的模凌两可问题不会爆发。因此你可以无忧无虑地从不同的namespaces汇入相同的符号（前提是你从未真正使用这个符号）。
